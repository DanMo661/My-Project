# 重构规则 (Refactoring Rules)

当用户说 "重构"、"重写"、"整理代码"、"优化代码"、"代码太乱/太臃肿" 时，遵循此流程。

---

## Phase 0: 确认用户意图

先澄清"重构"的具体含义：

- **仅重组**：保持相同功能和抽象水平，但按分层架构重新组织。添加结构体、提取辅助函数、修正命名。这是默认。
- **完全重写**：按本 skill 规范从头重写。仍保留原作者的抽象水平。

不确定就问，不要假设。

---

## Phase 1: 分析（不动代码）

### 1.1 代码水平评估（最关键）

| 水平 | 特征 | 你的改写要求 |
|------|------|-------------|
| 初学者 | 只用 `HAL_GPIO_*`、`HAL_I2C_*` 等库函数，不碰寄存器。可能复制粘贴，不懂 typedef | 只用 HAL/StdPeriph 库函数。不碰寄存器。保持模式简单。 |
| 中级 | 混用 HAL 和寄存器操作。有时用结构体。命名不一致 | 可以用寄存器简化代码。可以自由引入结构体。保留他们的外设库偏好。 |
| 高级 | 直接操作寄存器、CMSIS、DMA 链、内联汇编。代码"乱"是因为优化速度/大小 | 匹配他们的低层风格。不要把寄存器操作包装成 HAL 调用。 |

**评估方法**：Grep 寄存器访问模式（`GPIOA->`、`I2C1->`、`TIM3->`）。如果找不到寄存器访问，作者是库函数用户。

### 1.2 风格指纹 — 从原代码捕获：

- **命名规范**：拼音？CamelCase？snake_case？匈牙利命名？保留他们的规范
- **注释语言**：中文还是英文？匹配
- **变量作用域**：重度使用全局变量？那可能是他们的舒适区
- **缩进和括号风格**：K&R（开括号同行）vs Allman（开括号新行）
- **文件组织**：是否分离 .h/.c？是否有 config.h 模式？

### 1.3 读取项目文件并记录：

- 什么 MCU？用了哪些外设？
- 哪些文件是 CubeMX 生成的（只读）vs 手写的？
- 数据流是什么？传感器 → 哪里 → 输出到哪里？
- 哪些明显错误/坏了 vs 只是风格不同？

---

## Phase 2: 改写规则

**R1 — 匹配他们的抽象水平**
如果原代码用 `HAL_I2C_Master_Transmit(&hi2c1, ...)`，你也用同样的 HAL 调用。不要"升级"到 `I2C1->DR = data; while(!(I2C1->SR1 & I2C_SR1_TXE));`，即使更快。

**R2 — 每个函数都要有注释**
```c
/**
 * @brief  读取MPU6050的加速度数据
 * @param  dev   MPU6050设备结构体指针
 * @param  data  存放加速度数据的结构体指针
 * @retval HAL_OK 表示读取成功，其他值表示失败
 */
```
使用与原作者相同的注释语言。

**R3 — 非显而易见的逻辑要有行内注释**
魔法数字、位操作、时序变通、校准偏移 — 解释为什么。

**R4 — 遵循他们的命名，不是你的**
本 skill 的命名规范（Rule 2）适用于新项目。重构代码保留原作者的命名风格。

**R5 — 保持他们的依赖**
不要引入他们没有的新库或头文件。

**R6 — 分层（温和地）**
把散落在 main.c 的外设代码移到 Hardware/。但如果原项目没有 Hardware/ 目录且作者是初学者，用注释解释变化，不要静默重组。

---

## Phase 3: 验证和文档

改写后验证：
- [ ] 代码能用相同工具链编译
- [ ] 所有原始功能保留（无回归）
- [ ] 每个函数都有注释
- [ ] 抽象水平匹配原作者水平
- [ ] 命名在他们风格内一致

然后生成 `项目说明.md` — 见 `references/project-documentation.md`。
